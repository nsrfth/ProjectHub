---

## Goals

1. Admins can **create, edit, delete custom roles** per team.
2. Each role carries a set of **permissions** drawn from a fixed, code-bound
   list (~15 capabilities).
3. The existing `MANAGER` / `MEMBER` distinction is preserved as two
   **system roles** that auto-seed on every team. They're editable
   (admins can change their permission sets) but undeletable.
4. **Existing teams + members continue working unchanged** after the
   migration — zero behavioural drift.

## Non-goals

- Hierarchical roles / inheritance (rejected — adds complexity for
  minimal value).
- Multi-role assignment per user per team (one role per membership; flat).
- Instance-wide role templates (per-team is the chosen scope).
- Replacing the InstanceSetting policies from v1.18 (`tasks.dateEditRestriction`)
  and v1.21 (`trash.emptyAllowedRoles`). They stay as a **second layer** on
  top of the permission check.
- A "permissions" table. Permissions are a hardcoded constant list — adding
  a 16th requires a code change. That's deliberate; see "Pitfalls" below.

## Decisions already locked

| Decision | Value |
|---|---|
| Role scope | Per-team |
| Hierarchy | Flat (no inheritance) |
| Migration strategy | Auto-create `Manager` + `Member` system roles per team; backfill existing `TeamMembership` rows |
| Permission grain | ~15 capabilities matched to today's enforcement points |
| InstanceSetting v1.18 / v1.21 | Kept as a second gate ON TOP of the role permission |
| Code policy | `GlobalRole.ADMIN` bypasses every role check — lockout-proof escape hatch |

## Data model

Three tables added; one existing table extended.

### `Role` (new)

```prisma
model Role {
  id          String   @id @default(cuid())
  teamId      String
  name        String                              // unique per team
  description String?
  isSystem    Boolean  @default(false)            // built-in; editable but not deletable
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  team        Team     @relation(fields: [teamId], references: [id], onDelete: Cascade)
  permissions RolePermission[]
  memberships TeamMembership[] @relation("MembershipRole")

  @@unique([teamId, name])
  @@index([teamId])
}
```

### `RolePermission` (new)

```prisma
model RolePermission {
  roleId     String
  permission String   // matches a constant in code; service-layer validates

  role       Role     @relation(fields: [roleId], references: [id], onDelete: Cascade)

  @@id([roleId, permission])
}
```

### `TeamMembership` (extended)

```prisma
model TeamMembership {
  // ... existing fields ...

  // v1.23: replaces the legacy `role` enum (kept for one release as a
  // read-side compat shim; dropped in v1.24).
  roleId String?
  role_  TeamRole?              // ← rename of existing `role`; nullable now

  customRole Role? @relation("MembershipRole", fields: [roleId], references: [id], onDelete: SetNull)
}
```

**Why both columns coexist for one release**: drop-then-re-add is safer than
in-place rename, and it gives us a free fallback if the migration mis-maps.
The service-layer reads `roleId` first; falls back to `role_` if null.
v1.24 drops `role_`.

## Migration: `20260525000000_rbac`

```sql
-- 1. Create the tables.
CREATE TABLE "Role" (
  "id" TEXT PRIMARY KEY,
  "teamId" TEXT NOT NULL REFERENCES "Team"("id") ON DELETE CASCADE,
  "name" TEXT NOT NULL,
  "description" TEXT,
  "isSystem" BOOLEAN NOT NULL DEFAULT false,
  "createdAt" TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP,
  "updatedAt" TIMESTAMP(3) NOT NULL
);
CREATE UNIQUE INDEX "Role_teamId_name_key" ON "Role"("teamId", "name");
CREATE INDEX "Role_teamId_idx" ON "Role"("teamId");

CREATE TABLE "RolePermission" (
  "roleId" TEXT NOT NULL REFERENCES "Role"("id") ON DELETE CASCADE,
  "permission" TEXT NOT NULL,
  PRIMARY KEY ("roleId", "permission")
);

-- 2. Rename TeamMembership.role -> role_ (free up the name; deprecate).
ALTER TABLE "TeamMembership" RENAME COLUMN "role" TO "role_";

-- 3. Add roleId column (nullable until backfill is done).
ALTER TABLE "TeamMembership" ADD COLUMN "roleId" TEXT
  REFERENCES "Role"("id") ON DELETE SET NULL;

-- 4. For each team, create Manager + Member system roles.
INSERT INTO "Role" ("id", "teamId", "name", "description", "isSystem", "updatedAt")
SELECT
  'mgr_' || "id",
  "id",
  'Manager',
  'Default Manager role (system-managed)',
  true,
  CURRENT_TIMESTAMP
FROM "Team";

INSERT INTO "Role" ("id", "teamId", "name", "description", "isSystem", "updatedAt")
SELECT
  'mem_' || "id",
  "id",
  'Member',
  'Default Member role (system-managed)',
  true,
  CURRENT_TIMESTAMP
FROM "Team";

-- 5. Seed default permissions on the system roles (15 rows × N teams for
--    Manager; 2 rows × N teams for Member; see "Default role contents").
-- (SQL omitted for brevity — see migration file at write time.)

-- 6. Backfill TeamMembership.roleId.
UPDATE "TeamMembership" tm
SET "roleId" = (
  CASE tm."role_"
    WHEN 'MANAGER' THEN 'mgr_' || tm."teamId"
    WHEN 'MEMBER'  THEN 'mem_' || tm."teamId"
  END
);
```

The legacy `role_` column stays NULL-able + populated; service reads
`roleId` first, falls back to `role_`. v1.24 drops `role_`.

## Service layer

### New: `backend/src/services/rolesService.ts`

- `listRoles(teamId)`
- `getRole(teamId, roleId)`
- `createRole(teamId, { name, description, permissions })` — requires `team.manage_roles`
- `updateRole(teamId, roleId, patch)` — same; rejects rename of system roles' `name`
- `deleteRole(teamId, roleId)` — rejects if `isSystem` or if any membership uses it
- `setPermissions(teamId, roleId, permissions)` — validates each is in the constant list

### New: `backend/src/middleware/requirePermission.ts`

```ts
export function requirePermission(p: Permission): preHandlerHookHandler {
  return async (request) => {
    if (!request.user) throw Errors.unauthorized();
    if (request.user.globalRole === 'ADMIN') return;  // bypass

    const m = (request as any).membership as TeamMembership;
    if (!m) throw Errors.internal('Missing team membership context');

    const has = await prisma.rolePermission.findUnique({
      where: { roleId_permission: { roleId: m.roleId!, permission: p } },
    });
    if (!has) throw Errors.forbidden(`Missing permission: ${p}`);
  };
}
```

### Existing v1.18/v1.21 InstanceSetting layer stays untouched

- `tasks.dateEditRestriction = 'manager-only'` AND user lacks `task.modify_dates` permission → 403
- `trash.emptyAllowedRoles = 'admin'` AND user has only `trash.purge` permission (not global ADMIN) → 403

Both gates must pass for the operation. This is the "second layer" decision.

## API surface

New routes under `/api/teams/:teamId/roles`:

```
GET    /                — list roles in this team
POST   /                — create a custom role
GET    /:roleId         — get a role + its permissions
PATCH  /:roleId         — update name/description
PUT    /:roleId/permissions  — replace the permission set (idempotent)
DELETE /:roleId         — delete a custom role (rejects isSystem; rejects if in use)
```

All gated by `requirePermission('team.manage_roles')`.

The existing `PATCH /api/teams/:teamId/members/:userId` (today: `{ role: 'MANAGER' | 'MEMBER' }`) is extended to also accept `{ roleId: string }`. The two are mutually exclusive; if both supplied, 400. Legacy `role` is kept for one release for backwards-compat API callers.

New permission list endpoint (public-ish — needed by the matrix UI):

```
GET /api/system/permissions
  → { permissions: ['task.delete', 'task.modify_dates', ...],
      groups: { Tasks: [...], Comments: [...], ... } }
```

## Migration risks + mitigations

| Risk | Mitigation |
|---|---|
| Migration mis-maps a role; team locked out | Migration runs in a transaction; both `role_` and `roleId` coexist for one release; service reads `roleId` first, falls back to `role_` if null. Manual recovery: `UPDATE TeamMembership SET roleId = NULL WHERE teamId = X` ⇒ falls back to enum. |
| Admin revokes `team.manage_roles` from every role + isn't a global ADMIN themselves | `GlobalRole.ADMIN` always bypasses. Recovery path: any user can be promoted to global ADMIN via direct SQL by the operator. |
| Permission constant typo creates a "ghost" permission that's checked but never granted | Service layer validates every `RolePermission.permission` against `PERMISSIONS` constant on write. Old typos in DB don't grant anything (the check is exact-match). |
| User has both `roleId` set AND `role_` set, they disagree | `roleId` wins; `role_` is a fallback only when `roleId` is null. |
| System role's permissions get wiped by an admin | Allowed by design — the role can't be deleted, but its permissions can be edited (that's the whole point). If they wipe Manager's `team.manage_roles`, they need to be a global ADMIN to fix it. |

## Phase boundary

- One role per membership. Multi-role assignment is a v2 feature.
- 15 permissions. Adding a 16th requires a code change (deliberate).
- No "role templates" shared across teams. Each team defines its own roles. Copy-paste from team A to team B is a UI feature, not modelled in the schema.
- The legacy `TeamMembership.role_` column stays until v1.24.0. Operators should not write to it directly.
